# Audit complet SerieTime — 8 juillet 2026

> Audit à trois volets (sécurité serveur, qualité/UX mobile, écarts vs cahier des charges)
> réalisé après la mise en production. Les gravités sont calibrées pour le contexte
> « app entre amis », pas pour une banque. Chaque point garde sa référence fichier:ligne.

## Verdict global

Base saine : pas d'injection SQL possible (Prisma partout, zéro requête brute), validation
Zod sur toutes les routes, auth exigée globalement, isolation par compte correcte sur les
routes principales, aucune fuite de secrets dans les logs, erreurs 500 sans stack trace.
Infra prod stable (conteneur 141 Mo / 512, 0 redémarrage, backups nocturnes OK).
Les problèmes sont réels mais ciblés, et les fonctionnalités manquantes ont souvent
leur API déjà prête côté serveur (il ne manque que l'UI).

---

## Volet 1 — Sécurité serveur

### À corriger en priorité

1. **Aucun rate limiting** (`@fastify/rate-limit` absent) : brute-force illimité sur
   `/api/auth/login`, spam de comptes sur `/register` (`auth/routes.ts:158,184`).
   Bonus : `409 email_taken` permet d'énumérer les e-mails inscrits. **Gravité moyenne.**
2. **Zip bomb possible sur l'import TV Time** (`import-tvtime/service.ts:94-111`) :
   taille par entrée bornée (50 Mo) mais ni la somme décompressée ni le nombre
   d'entrées. Un ZIP piégé peut faire OOM le conteneur (512 Mo). **Moyenne.**
3. **`POST /api/backup/import` trop permissif** (`backup/routes.ts:82-117`) : upsert par
   id arbitraire sur les tables partagées (media/shows/seasons/episodes → un utilisateur
   peut écraser le catalogue commun) et réassignation possible de lignes d'autrui
   (statuts/listes). **Moyenne.**
4. **Imports TV Time non scopés par utilisateur** : le modèle `Import` n'a pas de
   `userId` (`schema.prisma:397`), toutes les routes `/api/import/tvtime/:id/*` sont
   accessibles inter-comptes (lecture du rapport + `confirm` des données d'autrui).
   **Moyenne.**

### À améliorer (basse gravité)

- Sessions : pas de purge des expirées, changement de mdp n'invalide pas les autres
  sessions (`auth/routes.ts:196-221`).
- `avatarUrl`/`coverUrl` non bornées (`profile/routes.ts:106-107`) : data-URL jusqu'à
  10 Mo stockée en DB → ajouter `.max()` (~500 Ko) + contrôle `data:image/`.
- Export de sauvegarde inclut `passwordHash` de l'appelant (`backup/routes.ts:15,30`).
- Mot de passe min 6 → passer à 8. bcrypt coût 10 → 12 possible.
- Pas de `@fastify/helmet` (ou headers côté Nginx). Prisma 5/Fastify 4 : un cycle majeur
  de retard, mise à jour à froid sans urgence.
- Aucune route de suppression de compte (RGPD) — le bouton existe dans l'app mais est mort.

### Conformes

Hash bcrypt correct, tokens 256 bits CSPRNG, CORS allowlist en prod, pas de routes debug,
pas de logs de secrets, Zip-Slip géré, erreurs 5xx opaques.

## Volet 2 — Qualité / UX de l'app mobile

### Gestion d'erreurs (le chantier n°1)

`LoadError` (+ RÉESSAYER) n'existe que sur À VOIR / À VENIR. Partout ailleurs, un échec
réseau donne un spinner infini ou un faux état vide :
- Spinner infini : profil (`(tabs)/profile.tsx:47`), fiche série (`show/[id].tsx:82`),
  épisodes (`show/[id].tsx:598`), profil public (`user/[id].tsx:51`), édition profil
  (`profile/edit.tsx:101`).
- Faux « vide » : films (`movies.tsx:37-57`, sans pull-to-refresh non plus),
  notifications (`notifications.tsx:56`), fil social (`social.tsx:71`), commentaires
  (`show/[id].tsx:843`), feed Explorer (message trompeur).
→ Extraire `LoadError` dans `components/ui.tsx` et brancher `isError`/`refetch` partout.

### Latence perçue (mutations non optimistes)

- **Coche épisode depuis À VOIR** (`(tabs)/index.tsx:58-64`) : aucun retour immédiat,
  incohérent avec la fiche série (déjà optimiste). Le plus fréquent des gestes.
- Vu/Pas vu film (`show/[id].tsx:55`), favori (`:51`), réactions emoji (`:762`),
  envoi de commentaire (`:744`).

### Web app (plateforme principale !)

- **« Partager » ne fait rien sur le web** : `Share.share()` non implémenté par
  react-native-web, erreur avalée (`show/[id].tsx:78-80`). → Web Share API + fallback
  copie. Aucun `Platform.OS` dans tout le code.
- `Alert.alert` quasi no-op sur web (`profile/edit.tsx:66,79`) → toast cross-platform.

### Bugs / culs-de-sac

- **Notifications → fiche film cassée** : `router.push('/show/id')` sans `?type=movie`
  (`notifications.tsx:68`).
- **Écran Import factice** (`import.tsx:18-46`) : chiffres codés en dur (42 séries...),
  boutons sans action.
- **Paramètres : boutons morts** : Modifier le mot de passe, Exporter mes données,
  Sauvegarde locale, Services d'abonnement, Choix des chaînes, toggle « Masquer les
  épisodes vus », et surtout « SUPPRIMER LE COMPTE » sans effet (`settings.tsx:44-100`).
- Sondage fiche série : réponses jamais persistées (état local perdu).
- Message technique visible utilisateur (« Renseignez TVDB_ENABLED... », `explore.tsx:170`).

### Performance

- **Liste d'épisodes non virtualisée** (`show/[id].tsx:601-673`) : ScrollView + .map,
  gel possible sur méga-saisons (One Piece) → SectionList. Le seul vrai risque perf.
- `Image` RN partout, jamais `expo-image` (pas de cache disque/placeholder) → migration M.
- Polling notifications 30 s permanent (`(tabs)/profile.tsx:40`).
- Composant géant : `show/[id].tsx` 875 lignes à éclater ; duplication follow/recherche
  utilisateurs (explore vs social vs user) ; `api.get<any>` sur la fiche.

## Volet 3 — Écarts vs cahier des charges / TV Time

> Note de cadrage : la spec interdisait le social, le projet l'a volontairement ajouté ;
> et la stack réelle est Expo (pas Vite+Capacitor). Les écarts ci-dessous sont évalués
> par rapport à l'objectif réel : répliquer TV Time pour une bande d'amis.

| Fonctionnalité | État | Importance | Remarque |
|---|---|---|---|
| Réactions par épisode | Absent (UI) | **Haute** | Colonne `reaction` en DB jamais alimentée ; le rituel TV Time n°1 |
| Notes d'épisodes + fiche/modal épisode (date de visionnage) | Partiel | **Haute** | Les API existent (`rating`, `date`), aucune UI |
| Écran Listes complet | Partiel | **Haute** | CRUD + réordonnancement serveur prêts, aucun écran mobile |
| Filtres/tris + bascule grille-liste | Partiel | Moyenne | Logique serveur prête (`movies/routes.ts:36-62`), pas d'UI |
| Écrans détail du profil (mes séries par statut, stats détaillées) | Absent | Moyenne | Chevrons du profil sans navigation |
| Statistique « temps films » | Partiel | Moyenne | Calculée serveur, carte manquante (3/4 affichées) |
| Badges / récompenses | Absent | Moyenne | Gros levier d'engagement de groupe |
| Thème sombre | Partiel | Moyenne | Le réglage existe mais n'a aucun effet (palette claire statique) |
| Mode hors-ligne (SW, file de mutations) | Absent | Moyenne | Pertinent vu les suspensions réseau iOS |
| Export/sauvegarde côté app | Partiel | Moyenne | API prête, boutons morts |
| Rewatch (revisionnages) | Absent | Basse/Moy. | Enum `rewatched` jamais émis |
| Invitation d'amis (lien), recherche par contacts | Absent | Moyenne | Amorçage du groupe |
| i18n, widgets, partage riche | Absent/Partiel | Basse | Groupe francophone, web app |

## Volet 4 — Infrastructure (vérifiée sur le VPS)

- Conteneur : stable, 141 Mo / 512, 0 restart. ✅
- Backups : cron 4 h 30 fonctionne (2 sauvegardes présentes). ✅
- **Logs Docker sans rotation** (`json-file` sans max-size) → à borner (ex. 10 Mo × 3).
- **Aucun header de cache Nginx sur la web app** : le bundle JS (~2 Mo, nom hashé donc
  cachable à vie) est retéléchargé inutilement → `expires 1y` sur `/_expo/` + assets.
- **Backups stockés sur le même disque que la prod** → copie hebdo hors VPS à prévoir.
- Pas de monitoring/alerte si le conteneur tombe (un simple healthcheck cron suffirait).
- Pas de CI GitHub (les 74 tests ne tournent qu'en local) → GitHub Action `pnpm test` sur PR.
- ApiCache : 210 entrées, pas de purge des expirées (croissance lente, à surveiller).

## Plan proposé (3 vagues)

**Vague 1 — Blindage (1 session)** : rate limiting auth, bornes zip/avatars, correctifs
backup-import et imports scopés, LoadError généralisé, coche optimiste À VOIR, Partager
web, bug notif film, rotation logs + cache Nginx + backup hors-site + CI.

**Vague 2 — Finitions TV Time (1-2 sessions)** : fiche/modal épisode (note, date,
réaction par épisode), écran Listes, filtres/tris + grille-liste, carte « temps films »,
écrans détail profil, brancher les boutons morts des Paramètres (dont suppression de
compte côté serveur + app).

**Vague 3 — Confort (au fil de l'eau)** : thème sombre réel, badges/récompenses,
lien d'invitation d'amis, expo-image, SectionList épisodes, éclatement de
`show/[id].tsx`, montée Prisma/Fastify, mode hors-ligne léger.
